# Security policy

## Reporting a vulnerability

If you find a security issue in Vitronitor, please report it privately rather
than opening a public issue. Use one of:

- GitHub's [private vulnerability reporting](https://docs.github.com/en/code-security/security-advisories/guidance-on-reporting-and-writing-information-about-vulnerabilities/privately-reporting-a-security-vulnerability)
  on this repo
- Email the maintainers (see the repo's `package.json` `author` field or the
  README for contact details)

We aim to acknowledge reports within 5 business days and ship a fix or
mitigation guidance within 30 days for high-severity issues.

## Scope

In scope:

- The boilerplate code in this repo (server, client, scripts, workflows).
- Documented patterns whose recommended implementation is unsafe.
- Default configurations that meaningfully weaken security.

Out of scope:

- Third-party dependencies — report those upstream (Supabase, Electric,
  Hono, Capacitor, Electron, etc.). We will pull in upstream fixes promptly.
- Forks and downstream deployments — contact the operator.
- Issues that require an attacker to already control the user's machine,
  Apple Developer account, Supabase project, or S3 credentials.

## Hardening checklist for downstream users

If you're shipping an app built on Vitronitor, the following are your
responsibility — the boilerplate cannot enforce them:

1. **Rotate the OTA signing key** generated by `scripts/setup-signing-key.sh`
   if it leaks. The same key signs both iOS Capgo and Electron renderer
   bundles. Replace the public key in [`capacitor.config.ts`](./capacitor.config.ts)
   and [`electron/main/renderer-ota.ts`](./electron/main/renderer-ota.ts), then
   issue a new shell binary so devices can adopt the rotated public key.
2. **Never commit `.env.local`, `.capgo_key_v2`, or `*.pem`** — they are
   gitignored, but verify before pushing.
3. **Set the Match cert repo (`MATCH_GIT_URL`) to a private repo you control**
   — never a public one. Encrypted match contents are not safe in public.
4. **Restrict the Supabase service-role key to server-side code only**.
   It bypasses RLS. The boilerplate uses it in `lib/supabase/admin.ts` —
   keep that import out of any module that ships to the browser.
5. **Audit RLS policies** for every new table. The notes example uses
   org-scoped policies as the canonical pattern — copy the shape, don't
   rely on `withAuth`'s WHERE injection alone.
6. **Pin GitHub Actions to commit SHAs** for third-party actions if you
   care about supply-chain integrity. The boilerplate uses `@v4` tags
   for first-party actions only.
7. **Set CSP origins** in `examples/server-hono/server/middleware/security-headers.ts` to your
   actual hosts. The default is permissive enough to develop against.
8. **Validate every `X-Org-Id` request server-side**. The middleware does
   this by default; do not bypass it for "internal" routes.
